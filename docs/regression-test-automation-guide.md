# リグレッションテスト全自動化 — 必要情報ガイド

> SaaS / Web アプリの自社製品を対象に、手動リグレッションテストを E2E で全自動化するために **事前に収集すべき情報** を条件別に整理したガイドです。

## 目的

- リグレッションテスト自動化プロジェクトの **立ち上げ前に必要な情報を漏れなく特定** する
- 情報不足のまま着手して手戻りが発生するリスクを防ぐ
- チーム内のヒアリング・合意形成のチェックリストとして使う

## 対象読者

- QA エンジニア / テスト自動化リード
- 開発マネージャー / プロダクトマネージャー
- 自動化導入を検討中のチーム

## 前提条件

- 対象は Web ブラウザで操作する SaaS / Web アプリケーション
- 現状は手動テスト（完全手動〜一部自動化）を想定
- 自動化のスコープは **テスト設計 → ケース管理 → 実行 → 判定 → レポート → CI/CD 統合** の E2E 全体

## A. プロダクト情報

プロダクトの技術的な構成を把握することで、テストフレームワークの選定や自動化の難易度を見積もる。

| # | 必要な情報 | なぜ必要か | 情報がない場合のリスク | 重要度 |
|---|-----------|-----------|---------------------|--------|
| A-1 | **アーキテクチャ構成**（SPA / SSR / MPA / ハイブリッド） | テストフレームワーク選定・待機戦略（SPA の非同期描画等）に直結する | 非同期処理の待機が不適切になり Flaky test が多発する | 必須 |
| A-2 | **フロントエンド技術スタック**（React / Vue / Angular / Web Components 等） | フレームワーク固有のセレクタ戦略・レンダリング特性の理解に必要 | 要素取得が不安定になりテストが壊れやすくなる | 必須 |
| A-3 | **バックエンド技術スタック**（言語・フレームワーク・API 方式） | API テスト併用時の設計、テストデータ準備方法に影響する | テストデータのセットアップが非効率になる | 推奨 |
| A-4 | **データベース構成**（RDBMS / NoSQL / 種類・バージョン） | テストデータのシード・クリーンアップ方法の設計に必要 | テスト間のデータ干渉が発生し結果が不安定になる | 推奨 |
| A-5 | **認証・認可方式**（ID/PW / SSO / OAuth / SAML / MFA） | ログイン自動化の設計に必要。MFA は特別な対応が必要 | ログイン処理が自動化できずテスト全体がブロックされる | 必須 |
| A-6 | **外部サービス連携**（決済 / メール / SMS / 地図 / AI 等） | モック・スタブ境界の設計に必要 | 外部障害でテストが失敗する、またはコストが発生する | 必須 |
| A-7 | **マルチテナント構成** の有無と方式 | テナント間のデータ分離・テストアカウント設計に影響する | テナント間でデータが汚染され本番影響のリスクがある | 推奨 |
| A-8 | **対応ブラウザ・デバイス**（Chrome / Firefox / Safari / モバイル） | クロスブラウザテストの範囲とツール選定に必要 | 特定ブラウザでの不具合を見逃す | 必須 |
| A-9 | **デプロイ方式**（コンテナ / サーバーレス / PaaS） | テスト環境構築の自動化方針に影響する | テスト環境のセットアップに時間がかかる | 任意 |
| A-10 | **URL 構造・ルーティング**（ハッシュ / ヒストリー / 動的パス） | ページ遷移の検証方法・ナビゲーション待機に影響する | 遷移検証が不正確になる | 推奨 |

## B. 既存テスト資産

現在のテスト資産を棚卸しすることで、自動化の出発点と優先順位を決める。

| # | 必要な情報 | なぜ必要か | 情報がない場合のリスク | 重要度 |
|---|-----------|-----------|---------------------|--------|
| B-1 | **手動テストケース一覧**（テスト仕様書・手順書） | 自動化対象の母数を把握し、工数見積もりの基礎になる | スコープが不明確で見積もりが大幅にずれる | 必須 |
| B-2 | **テストケースの分類**（機能別 / 画面別 / 優先度別） | 自動化の優先順位付けに使う | 重要度の低いテストから着手してしまい ROI が悪化する | 必須 |
| B-3 | **テストケースの総数と実行時間**（手動での所要時間） | 自動化による時間削減効果の試算に必要 | 自動化の投資対効果を説明できない | 推奨 |
| B-4 | **過去の不具合データ**（バグが多い機能・画面） | リスクベースで自動化の優先順位を決められる | バグが頻発する箇所の自動化が後回しになる | 推奨 |
| B-5 | **テスト実行頻度・タイミング** | 自動化の実行スケジュール設計に必要 | 不適切なタイミングでの実行や実行漏れが起きる | 必須 |
| B-6 | **既存の自動テスト**（ユニット / API / E2E）の有無と範囲 | 重複を避け、E2E で補うべき範囲を特定する | 既にカバーされている箇所を二重にテストして無駄が生じる | 推奨 |
| B-7 | **テスト結果の記録方法**（スプレッドシート / ツール / なし） | テスト管理ツール導入の要否判断に使う | 結果の追跡・比較ができず品質トレンドが見えない | 任意 |

## C. 自動化対象の選定基準

すべてを自動化するのは非現実的。何を自動化し、何を手動で残すかの判断基準を整理する。

| # | 必要な情報 | なぜ必要か | 情報がない場合のリスク | 重要度 |
|---|-----------|-----------|---------------------|--------|
| C-1 | **ビジネスクリティカルな機能の一覧** | 最優先で自動化すべき機能を特定する | 収益に直結する機能のリグレッションを見逃す | 必須 |
| C-2 | **変更頻度の高い機能・画面** | 頻繁に壊れる箇所を優先的に自動化する | 手動テストのボトルネックが解消されない | 必須 |
| C-3 | **自動化が困難な機能の特定**（CAPTCHA / 地図操作 / ファイルアップロード / PDF 等） | 技術的な難易度と代替手段を事前に検討する | 自動化不可能な箇所で工数を浪費する | 必須 |
| C-4 | **手動で残すべきテストの基準**（探索的テスト / UX 評価 / アクセシビリティ） | 自動化のスコープを明確にし、期待値を合わせる | 「全自動化」への過度な期待が生まれ失望につながる | 推奨 |
| C-5 | **自動化 ROI の算出基準**（実行頻度 × 手動工数 × 不具合リスク） | 投資判断・優先順位付けの定量的根拠になる | 経営層やステークホルダーへの説明が困難になる | 推奨 |

## D. テスト設計情報

テストシナリオを設計するために必要なプロダクトの仕様・振る舞いの情報。

| # | 必要な情報 | なぜ必要か | 情報がない場合のリスク | 重要度 |
|---|-----------|-----------|---------------------|--------|
| D-1 | **画面一覧・画面遷移図** | テストシナリオのカバレッジを網羅的に設計する基礎になる | テスト漏れが発生する | 必須 |
| D-2 | **主要ユーザーフロー**（ハッピーパス） | 最優先で自動化すべきシナリオを定義する | ビジネス上重要なフローがテストされない | 必須 |
| D-3 | **ユーザーロール・権限マトリクス** | ロールごとの表示・操作制限のテストに必要 | 権限バグ（本来見えないデータが見える等）を見逃す | 必須 |
| D-4 | **入力バリデーションルール**（文字数 / 形式 / 必須項目） | 異常系テストデータの設計に必要 | バリデーション漏れの不具合を検知できない | 推奨 |
| D-5 | **状態遷移**（注文ステータス / ワークフロー等） | 状態遷移パスの網羅的なテスト設計に必要 | 特定の状態遷移パスでのバグを見逃す | 推奨 |
| D-6 | **テストデータのパターン**（正常系 / 異常系 / 境界値） | データ駆動テストの設計に必要 | テストカバレッジが不十分になる | 推奨 |
| D-7 | **環境依存条件**（タイムゾーン / 言語 / 通貨 / 地域） | 国際化対応のテスト範囲を決める | 特定地域・言語での不具合を見逃す | 任意 |
| D-8 | **非機能要件**（レスポンスタイム閾値 / 同時接続数） | パフォーマンスリグレッションの検知基準に使う | パフォーマンス劣化に気づかない | 任意 |

## E. テスト基盤・ツール選定

適切なツールを選定するために必要な技術要件と制約の情報。

| # | 必要な情報 | なぜ必要か | 情報がない場合のリスク | 重要度 |
|---|-----------|-----------|---------------------|--------|
| E-1 | **E2E テストフレームワークの候補と選定基準** | チームのスキル・プロダクト特性に合ったツールを選ぶ | 後から乗り換えが必要になり大幅な手戻りが発生する | 必須 |
| E-2 | **テスト管理ツール**（TestRail / Zephyr / Qase / スプレッドシート等） | テストケースの管理・実行履歴・レポートの基盤を決める | ケース管理が属人化し、追跡・共有ができなくなる | 推奨 |
| E-3 | **CI/CD ツール**（GitHub Actions / Jenkins / CircleCI / GitLab CI 等） | テスト実行の自動トリガー・パイプライン設計に必要 | 手動実行のままでは「全自動化」にならない | 必須 |
| E-4 | **テスト実行環境**（ローカル / CI サーバー / クラウド / BrowserStack 等） | 実行環境の構築・コスト見積もりに必要 | 実行環境の制約でテストが動かない | 必須 |
| E-5 | **並列実行・分散実行の要件** | テスト実行時間の短縮戦略に必要 | テストスイートが肥大化した際に実行時間が許容範囲を超える | 推奨 |
| E-6 | **レポート・可視化ツール**（Allure / HTML Reporter / ダッシュボード） | テスト結果の共有・分析手段を決める | 結果が見づらく、失敗の分析に時間がかかる | 推奨 |
| E-7 | **ビジュアルリグレッションテスト** の要否 | UI の見た目の変化を検知するかどうかを決める | CSS 変更による意図しないレイアウト崩れを見逃す | 任意 |

## F. テスト環境・データ

安定した自動テスト実行のために必要な環境とデータ管理の情報。

| # | 必要な情報 | なぜ必要か | 情報がない場合のリスク | 重要度 |
|---|-----------|-----------|---------------------|--------|
| F-1 | **テスト専用環境の有無と構成**（ステージング / QA 環境） | テストが本番に影響しない独立環境が必要 | 本番データの破損や意図しない副作用が発生する | 必須 |
| F-2 | **環境のプロビジョニング方法**（手動 / IaC / コンテナ） | テスト環境の再現性・構築速度に影響する | 環境差異によるテスト失敗が頻発する | 推奨 |
| F-3 | **テストデータのセットアップ方法**（API / DB 直接 / UI 操作 / シード） | 各テストの前提条件を効率的に準備する方法を決める | テストデータ準備に時間がかかりテストが遅くなる | 必須 |
| F-4 | **テストデータのクリーンアップ方針**（テストごと / スイートごと / 環境リセット） | テスト間の独立性を保証する | テスト実行順序に依存する不安定なテストになる | 必須 |
| F-5 | **外部サービスのモック・スタブ方針** | 外部依存を排除して安定性を確保する | 外部サービスの障害・レート制限でテストが失敗する | 必須 |
| F-6 | **機密データの扱い**（本番データのマスキング / 合成データ生成） | テストデータにおける個人情報保護の方針を決める | コンプライアンス違反のリスクがある | 推奨 |
| F-7 | **DB マイグレーション・スキーマ変更への追従方法** | スキーマ変更時にテストデータが壊れないようにする | DB 変更のたびにテストデータの修正が必要になる | 推奨 |

## G. CI/CD 統合

テストをパイプラインに組み込み、継続的に実行するための情報。

| # | 必要な情報 | なぜ必要か | 情報がない場合のリスク | 重要度 |
|---|-----------|-----------|---------------------|--------|
| G-1 | **テスト実行のトリガー条件**（PR 作成 / マージ / スケジュール / 手動） | いつテストを実行するかを決める | 適切なタイミングでテストが実行されず不具合が流出する | 必須 |
| G-2 | **パイプラインのステージ構成**（ビルド → ユニット → E2E → デプロイ） | テストの実行位置と依存関係を設計する | E2E テストがボトルネックになりデプロイが遅延する | 必須 |
| G-3 | **テスト実行時間の目標値** | 並列化・テスト分割の判断基準になる | テストが長すぎて開発サイクルが遅くなる | 推奨 |
| G-4 | **失敗時のブロック方針**（マージブロック / 警告のみ / Flaky 判定） | テスト失敗時のアクションを決める | Flaky test でマージがブロックされ開発速度が低下する | 必須 |
| G-5 | **テスト結果の通知先・方法**（Slack / メール / ダッシュボード） | 失敗をすばやく検知し対応するために必要 | テスト失敗に気づかず放置される | 推奨 |
| G-6 | **テストの分割・並列実行戦略**（シャーディング / タグベース） | 実行時間短縮のための構成を設計する | スイート肥大化時にパイプラインが実用的でなくなる | 推奨 |
| G-7 | **リトライ・再実行ポリシー**（自動リトライ回数 / 手動再実行フロー） | Flaky test による偽陽性を軽減する | 不安定なテストがチームの信頼を損なう | 推奨 |
| G-8 | **成果物の保存方針**（スクリーンショット / 動画 / トレース / ログ） | 失敗時のデバッグを効率化する | テスト失敗の原因調査に時間がかかる | 推奨 |

## H. 運用・保守

自動テストを長期的に運用するために必要な設計・体制の情報。

| # | 必要な情報 | なぜ必要か | 情報がない場合のリスク | 重要度 |
|---|-----------|-----------|---------------------|--------|
| H-1 | **テストコードの設計パターン**（Page Object Model / Screenplay 等） | 保守性の高いテストコードを書くための指針を決める | テストコードがスパゲッティ化し保守不能になる | 必須 |
| H-2 | **セレクタ戦略**（data-testid / aria-label / CSS / XPath） | UI 変更に強い要素特定方法を統一する | 小さな UI 変更でテストが大量に壊れる | 必須 |
| H-3 | **テストコードの保守体制・担当** | 誰がテストを書き、誰が壊れたテストを直すかを決める | テストが壊れたまま放置され無価値になる | 必須 |
| H-4 | **Flaky test 対策方針**（検知 / 隔離 / 修正フロー） | 不安定なテストへの対処ルールを事前に決める | Flaky test が蓄積しテスト結果への信頼が失われる | 必須 |
| H-5 | **テストコードのレビュー基準** | テストコードの品質を維持する | 低品質なテストが蓄積し技術的負債になる | 推奨 |
| H-6 | **新機能追加時のテスト追加フロー** | 開発プロセスにテスト追加を組み込む | 新機能のリグレッションテストが抜け落ちる | 必須 |
| H-7 | **テストのバージョン管理方針**（プロダクトコードと同一リポジトリ / 別リポジトリ） | テストとプロダクトの同期方法を決める | バージョン不整合でテストが動かなくなる | 推奨 |
| H-8 | **依存ライブラリのアップデート方針** | テストフレームワーク・ブラウザのバージョンアップ対応を計画する | セキュリティ脆弱性の放置や互換性問題が発生する | 任意 |

## I. 組織・プロセス

技術だけでなく、チーム体制やプロセスの情報も成功には不可欠。

| # | 必要な情報 | なぜ必要か | 情報がない場合のリスク | 重要度 |
|---|-----------|-----------|---------------------|--------|
| I-1 | **QA チームの体制・人数** | 自動化の推進リソースを見積もる | リソース不足で自動化が進まない | 必須 |
| I-2 | **チームのスキルセット**（プログラミング / テストツール経験） | トレーニング計画・ツール選定に影響する | 学習コストが高すぎてツールが活用されない | 必須 |
| I-3 | **開発チームとの連携フロー**（同一チーム / 別チーム / 外注） | テストコードの作成・レビュー・修正のワークフローを設計する | 開発と QA の分断でテストが陳腐化する | 必須 |
| I-4 | **リリースサイクル**（週次 / 隔週 / 月次 / 随時） | テスト実行頻度と自動化の緊急度を判断する | リリース頻度に対してテスト実行が追いつかない | 推奨 |
| I-5 | **テスト自動化の成熟度**（初めて / 過去に失敗経験あり / 部分導入済み） | 導入アプローチとリスク対策を調整する | 過去の失敗と同じ轍を踏む | 推奨 |
| I-6 | **教育・トレーニング計画** | チーム全体の自動化スキルを底上げする | 特定の人に依存し、その人が抜けると崩壊する | 推奨 |
| I-7 | **導入スケジュール・マイルストーン** | 段階的導入の計画と進捗管理に使う | ゴールが曖昧で完了判定ができない | 推奨 |
| I-8 | **ステークホルダーの期待値**（経営層 / PM / 開発チーム） | 自動化で達成すべき目標を合意する | 過度な期待や誤解により評価が不当に低くなる | 推奨 |
| I-9 | **予算・コスト制約**（ツールライセンス / クラウド実行費用 / 人件費） | ツール選定と実行環境の制約条件になる | コスト超過でプロジェクトが中断する | 必須 |
